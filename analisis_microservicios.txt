ANÁLISIS DE MICROSERVICIOS - LeaseFlow (Rentify)
Fecha: 2026-05-21

NOTA IMPORTANTE
- Este documento describe servicios, capacidades, reglas de negocio, métodos HTTP y formatos de intercambio.
- No se listan rutas/endpoints concretos (según lo solicitado). Se describe “qué hace” cada API y “cómo intercambia” datos.
- No se incluyen credenciales/secretos aunque existan en configuración local del repo.

====================================================================
1) VISIÓN GENERAL DEL SISTEMA
====================================================================
El repositorio contiene 6 microservicios Spring Boot (Java 17, Spring Boot 3.5.x) empaquetados con Dockerfile y desplegados individualmente (pipeline CI/CD actualiza Azure Container Apps por servicio).

Servicios:
1. userservice      (identidades, roles, estados, login)
2. propertyservice  (propiedades + catálogos geográficos + fotos)
3. documentService  (documentos de usuario + estados/tipos de documento)
4. applicationService (solicitudes de arriendo + registros de arriendo)
5. contactService   (mensajes de contacto + respuesta/admin + estadísticas)
6. reviewService    (reseñas + promedios + tipos de reseña)

Características transversales comunes:
- Estilo de API: REST sobre HTTP (Spring MVC) con payloads JSON (application/json) y validaciones Jakarta Validation.
- Comunicación servicio-a-servicio: HTTP interno usando WebClient (Spring WebFlux) con timeouts y manejo de errores “tolerante” (fallback a null/false en algunos casos).
- Persistencia: MySQL (instancia administrada) con un esquema/BD distinto por servicio (users, properties, documents, applications, contact, reviews).
- ORM: Spring Data JPA + Hibernate, estrategia DDL “update” (evoluciona el schema sin borrar datos).
- Serialización: Jackson. En algunos servicios se controlan fechas (por ejemplo: formato y zona horaria).
- Observabilidad: logs por servicio (niveles configurables), Swagger/OpenAPI (springdoc) habilitado en la mayoría, Actuator expuesto (al menos health/info; en algunos también metrics).
- CORS: configurado por servicio vía CorsConfig (enfoque de desarrollo local).

====================================================================
2) PUNTOS DE CONEXIÓN (TOPOLOGÍA DE INTEGRACIÓN)
====================================================================
2.1 Persistencia (DB)
- Cada microservicio usa su propia base de datos MySQL (patrón “database per service”).
- Conexión con SSL habilitada.
- Tablas generadas/actualizadas por Hibernate (ddl-auto=update).

2.2 Comunicación entre microservicios (HTTP + JSON)
No hay service discovery ni gateway en el repositorio: cada servicio usa URLs base configurables mediante properties del tipo:
- microservices.user-service.url
- microservices.property-service.url
- microservices.document-service.url
- microservices.application-service.url
- microservices.review-service.url
- microservices.contact-service.url

Patrón técnico común en los “clients”:
- WebClient.Builder -> método HTTP (principalmente GET) -> retrieve() -> bodyToMono(tipo) -> timeout -> onErrorResume -> block().
- Manejo de fallos:
  - Algunos clientes convierten errores a “Mono.empty()” y devuelven null, o devuelven false/0 ante fallos.
  - En errores “críticos” a veces se lanza MicroserviceException.

2.3 Mapa de dependencias (quién llama a quién)
- applicationService
  - llama a userservice (validar usuario/rol)
  - llama a propertyservice (validar existencia/“disponibilidad” de propiedad)
  - llama a documentService (verificar documentos aprobados)
  - (tiene properties para review/contact pero no se observan dependencias directas en la lógica principal mostrada)

- documentService
  - llama a userservice (validar existencia de usuario y rol; expandir detalles de usuario en respuestas)

- contactService
  - llama a userservice (validar existencia, validar rol admin, expandir datos de usuario)

- reviewService
  - llama a userservice (validar existencia/rol; expandir usuario y usuario reseñado)
  - llama a propertyservice (validar propiedad; evitar reseña a propiedad propia; expandir propiedad)

- propertyservice
  - en la lógica principal no se observa invocación obligatoria a otros servicios; opera sobre su propio dominio y datos.

- userservice
  - expone datos base (usuarios/roles/estados) consumidos por los demás. No depende funcionalmente de otros microservicios.

3) SERVICIOS (DESGLOSE DETALLADO)

--------------------------------------------------------------------
3.1 userservice (Gestión de Usuarios, Roles, Estados y Login)
--------------------------------------------------------------------
Responsabilidad principal
- Fuente de verdad de identidad del usuario: alta, consulta, actualización, cambios de rol/estado y gamificación (puntos).
- Catálogos maestros del dominio “usuarios”: roles y estados.

Persistencia y modelo de datos
- Entidades principales: Usuario, Rol, Estado.
- Campos y reglas destacadas (por DTO y servicio):
  - Usuario: nombres, rut, email, teléfono, fecha nacimiento, clave (hash), rolId, estadoId, puntos, duocVip, código de referido, fechas de creación/actualización.
  - Rol: ADMIN / PROPIETARIO / ARRIENDATARIO.
  - Estado: ACTIVO / INACTIVO / SUSPENDIDO.

Capacidades funcionales (qué permite hacer)
- Registro de usuario con validaciones:
  - Fecha nacimiento obligatoria en formato “yyyy-MM-dd”.
  - Edad mínima: 18 años.
  - Email y RUT únicos.
  - Generación de código de referido único (con reintentos).
  - Asignación por defecto de rol ARRIENDATARIO si no se especifica rolId.
  - Marcado DUOC VIP si el email termina en “@duoc.cl”.
  - Puntos iniciales: 0.
- Login:
  - Autenticación por email + clave (BCryptPasswordEncoder).
  - Rechaza login si el estado del usuario es INACTIVO o SUSPENDIDO.
  - Devuelve un “LoginResponse” con mensaje + datos del usuario (no se observa generación de token).
- Consultas:
  - Listado de usuarios y consulta individual por ID/email/rol.
  - Listado de usuarios DUOC VIP.
  - Verificación de existencia (boolean).
- Administración y mantenimiento:
  - Actualización parcial (vía DTO de update) para datos de usuario, rol y estado.
  - Cambios puntuales de rol, estado y puntos mediante operaciones tipo PATCH.
- Catálogos:
  - CRUD de roles (crear/listar/obtener).
  - CRUD de estados (crear/listar/obtener).

Métodos HTTP utilizados (a nivel de capacidades)
- POST: alta de usuario; login; creación de roles; creación de estados.
- GET: listados y lecturas (usuarios, roles, estados) + verificaciones de existencia.
- PUT: actualización de usuario (admin).
- PATCH: cambio de rol/estado/puntos.

Formatos de entrada/salida
- Entrada principal: JSON (application/json) con validaciones Jakarta:
  - UsuarioDTO (registro/lectura): incluye campos obligatorios y campos de solo lectura; fechas como string “yyyy-MM-dd”.
  - LoginDTO (email + clave).
  - UsuarioUpdateDTO (actualización parcial administrativa).
  - RolDTO/EstadoDTO para catálogos.
- Salida: JSON con DTOs. Existe parámetro includeDetails que habilita expansión de rol/estado embebidos en el UsuarioDTO.

Seguridad y autorización (estado actual)
- Seguridad HTTP: SecurityFilterChain configurado como stateless, CSRF deshabilitado, y “permitAll” a todas las solicitudes.
- Autenticación/Autorización real se basa en lógica de negocio (por ejemplo, estados en login) y/o en que otros servicios consulten el rol.
- No se observa emisión/validación de JWT ni sesiones; el login es “validación de credenciales” que retorna datos.

Observabilidad y soporte
- Swagger/OpenAPI habilitado (springdoc).
- Logging con nivel DEBUG en el paquete del servicio.
- Perfil dev activo por defecto y auto-carga de datos (maestros y test) habilitable por properties.

Dependencias hacia otros servicios
- No se observan dependencias salientes obligatorias.

3.2 propertyservice (Propiedades + Catálogos + Fotos)
--------------------------------------------------------------------
Responsabilidad principal
- Gestión del inventario de propiedades para arriendo.
- Catálogos territoriales y de clasificación (regiones, comunas, tipos, categorías).
- Gestión de fotos asociadas a propiedades, incluyendo upload y ordenamiento.

Persistencia y modelo de datos
- Entidades principales: Property, Foto, Tipo, Comuna, Region, Categoria.
- Relación:
  - Una propiedad referencia Tipo y Comuna; Comuna referencia Region.
  - Una propiedad puede tener múltiples fotos y categorías.

Capacidades funcionales
Propiedades (core):
- Crear propiedad:
  - Código único (no duplicado).
  - Divisa válida: CLP, USD, EUR.
  - Precio mensual > 0.
  - m2 dentro de rango [1..10000].
  - Habitaciones dentro de [0..50], baños dentro de [0..20].
  - Requiere tipoId, comunaId y propietarioId.
  - Fecha de creación se asigna si no viene informada.
- Consultas:
  - Listado general con “includeDetails” (expande tipo/comuna/región, fotos y categorías).
  - Consulta por ID y por código.
  - Listado de propiedades por usuario propietario (propietarioId).
  - Búsqueda con filtros opcionales (comuna, tipo, rango de precios, habitaciones, baños, petFriendly).
  - Verificación de existencia (boolean).
- Actualizar propiedad:
  - Permite cambios parciales (código, título, precio, divisa, m2, nHabit, nBanos, petFriendly, dirección, tipo, comuna, propietarioId).
  - Re-valida duplicidad de código si se cambia.
- Eliminar propiedad:
  - Borrado por ID.

Fotos:
- Subida de imagen asociada a una propiedad:
  - Formatos permitidos por content-type: image/jpeg, image/jpg, image/png, image/webp.
  - Tamaño máximo: 10 MB.
  - Límite: máximo 20 fotos por propiedad.
  - Almacenamiento: filesystem local bajo un directorio configurable (por defecto “uploads/properties/{propertyId}/”).
  - Persistencia: se guarda metadata de foto (nombre, “url” como ruta local, sortOrder) en BD.
- Listado de fotos por propiedad (ordenado por sortOrder).
- Obtener foto por ID (metadata).
- Eliminar foto:
  - Elimina el registro en BD.
  - Intenta borrar el archivo del disco (si falla, continúa con eliminación en BD).
- Reordenar fotos:
  - Recibe una lista JSON de IDs de fotos con el orden deseado y reescribe sortOrder.

Catálogos (regiones/comunas/tipos/categorías)
- Operaciones CRUD típicas sobre cada catálogo.
- Conversión entidad<->DTO con ModelMapper (en varios controladores se accede directo al repository).

Métodos HTTP utilizados
- Propiedades: POST/GET/PUT/DELETE + GET de búsqueda con filtros.
- Fotos: POST (multipart), GET, PUT (reorden), DELETE.
- Catálogos: POST/GET/PUT/DELETE.

Formatos de entrada/salida
- JSON (application/json) para propiedades y catálogos.
  - PropertyDTO: incluye BigDecimal para precios y m2; LocalDate para fecha (serializa “yyyy-MM-dd”).
  - Particularidad: nHabit y nBanos están modelados como int y usan @JsonProperty/@JsonAlias para compatibilidad con distintos clientes.
- multipart/form-data para subir fotos: parámetro “file” como archivo binario.
- Respuestas: JSON con DTOs; cuando includeDetails=true, se adjuntan objetos embebidos (TipoDTO, ComunaDTO con RegionDTO, lista de FotoDTO, lista de CategoriaDTO).

Integraciones
- No se observa dependencia funcional saliente obligatoria.
- Otros servicios consumen propertyservice para: verificar existencia, obtener datos de propiedad y evitar reseñas a propiedad propia.

Observabilidad y soporte
- Actuator expuesto (health/info/metrics en configuración).
- Swagger/OpenAPI habilitado.
- Profile dev con auto-población de datos (maestros y test).

3.3 documentService (Documentos de Usuario + Estados/Tipos de Documento)
--------------------------------------------------------------------
Responsabilidad principal
- Registrar y gestionar documentos asociados a usuarios para procesos de verificación (ej. identidad, antecedentes, liquidaciones).
- Mantener estados de documento (pendiente/aceptado/rechazado/en revisión) y tipos de documento.

Persistencia y modelo de datos
- Entidades principales: Documento, Estado, TipoDocumento.
- Documento incluye:
  - usuarioId (referencia externa al userservice)
  - estadoId y tipoDocId (catálogos internos)
  - nombre del documento
  - fechaSubido y fechaActualizacion
  - observaciones (motivo de rechazo) y revisadoPor (id admin)

Capacidades funcionales
Documentos:
- Crear documento:
  - Valida que el usuario existe consultando userservice.
  - Valida que el rol del usuario permite subir documentos (ARRIENDATARIO, PROPIETARIO o ADMIN).
  - Enforce de límite: máximo 10 documentos por usuario.
  - Valida que existan estadoId y tipoDocId en su BD.
  - Genera fechaSubido automáticamente.
  - Nota: aunque se describe como “subir”, en la implementación actual el input es un DocumentoDTO (JSON) con “nombre” y referencias; no hay upload binario implementado en el controlador.
- Consultas:
  - Listado de documentos (con includeDetails opcional).
  - Consulta de documento por ID.
  - Listado de documentos por usuario (valida existencia del usuario vía userservice).
- Verificación de aprobados:
  - Operación que determina si un usuario tiene al menos un documento con estado “ACEPTADO”.
  - Mantiene un endpoint alternativo marcado deprecated (a nivel de código).
- Actualización de estado:
  - Cambio de estado “simple” (sin observaciones).
  - Cambio de estado “con observaciones”:
    - Si el nuevo estado es RECHAZADO, observaciones es obligatorio.
    - Permite registrar revisadoPor (ID del admin) y motivo.
- Eliminación de documento por ID.

Catálogos internos:
- Tipos de documento:
  - Listado, consulta por ID, creación, actualización y eliminación.
  - Tipos predefinidos conceptuales: DNI, PASAPORTE, LIQUIDACIÓN SUELDO, etc.
- Estados:
  - (Controlador/servicio equivalente) operaciones típicas de mantenimiento de estados.

Métodos HTTP utilizados
- POST: crear documento; crear tipo de documento; crear estado.
- GET: listados/lecturas; verificación de aprobados.
- PATCH: actualización de estado (simple y con observaciones).
- PUT: actualización de tipo de documento / estado (si aplica en su controlador).
- DELETE: eliminación de documento / tipo / estado.

Formatos de entrada/salida
- JSON (application/json) como formato principal:
  - DocumentoDTO: nombre (máx 60), usuarioId, estadoId, tipoDocId; observaciones (máx 500).
  - ActualizarEstadoRequest: estadoId, observaciones (opcional/obligatorio según estado), revisadoPor.
- Respuestas:
  - DTOs enriquecidos cuando includeDetails=true: agrega estadoNombre, tipoDocNombre y objeto UsuarioDTO (consultado en userservice).
- Fechas:
  - DocumentoDTO usa java.util.Date; el servicio configura Jackson para evitar timestamps y usar zona horaria específica.

Integraciones
- Dependencia directa: userservice (validación de existencia, obtención de rol y datos del usuario).
- Consumidores típicos:
  - applicationService llama para verificar documentos aprobados.

Observabilidad y soporte
- Actuator habilitado (incluye health/info/metrics según properties).
- Inicialización SQL (data.sql) configurada para poblar catálogos en arranque, ejecutando después de Hibernate.

3.4 applicationService (Solicitudes de Arriendo + Registros de Arriendo)
--------------------------------------------------------------------
Responsabilidad principal
- Orquestar el flujo de “postulación/solicitud” de arriendo y el “registro” de arriendos activos.
- Centraliza reglas de negocio que requieren validar usuario, propiedad y documentos.

Persistencia y modelo de datos
- Entidades principales: SolicitudArriendo, RegistroArriendo.
- SolicitudArriendo:
  - usuarioId (externo)
  - propiedadId (externo)
  - estado (PENDIENTE / ACEPTADA / RECHAZADA)
  - fechaSolicitud
- RegistroArriendo:
  - solicitudId (interno; referencia a SolicitudArriendo)
  - fechaInicio, fechaFin (opcional), montoMensual
  - activo (boolean)

Capacidades funcionales
Solicitudes de arriendo:
- Crear solicitud con validaciones:
  - Usuario debe existir (consulta userservice) y su rol debe permitir crear solicitudes:
    - Permitidos: ARRIENDATARIO y ADMIN (por rolId).
  - Límite: máximo 3 solicitudes activas (en estado PENDIENTE) por usuario.
  - Evita duplicidad: una solicitud pendiente por (usuario, propiedad).
  - Propiedad debe existir (consulta propertyservice).
  - “Disponibilidad” de propiedad:
    - En la implementación actual, se asume disponible si la propiedad existe (placeholder para futura lógica).
  - Documentos aprobados:
    - Verifica en documentService si el usuario tiene documentos aprobados.
    - Si falla, bloquea la creación.
  - Estado inicial: PENDIENTE.
  - Fecha de solicitud: asignada automáticamente.
- Consultas:
  - Listado general con includeDetails (expande usuario y propiedad llamando a sus servicios).
  - Consulta por ID con includeDetails.
  - Listado por usuario y por propiedad.
- Cambio de estado:
  - Actualiza estado por string, validando que sea uno de los estados permitidos.

Registros de arriendo:
- Crear registro:
  - Solicitud debe existir.
  - Solo se puede crear registro si la solicitud está en estado ACEPTADA.
  - Evita duplicidad: no permite más de un registro activo por solicitud.
  - Valida coherencia de fechas si hay fechaFin (inicio no puede ser posterior a fin).
  - Marca activo=true al crear.
- Consultas:
  - Listado general con includeDetails (expande la solicitud; y la solicitud puede expandir usuario y propiedad).
  - Consulta por ID.
  - Listado por solicitud.
- Finalizar registro:
  - Cambia activo=false y setea fechaFin a la fecha actual.

Métodos HTTP utilizados
- POST: crear solicitud; crear registro.
- GET: listados/lecturas.
- PATCH: cambio de estado de solicitud; finalización de registro.

Formatos de entrada/salida
- JSON (application/json):
  - SolicitudArriendoDTO: usuarioId, propiedadId; estado/fecha en solo lectura; posibilidad de usuario y propiedad embebidos con includeDetails.
  - RegistroArriendoDTO: solicitudId, fechaInicio, fechaFin (opcional), montoMensual; activo y solicitud embebida en solo lectura.
- Parámetros de consulta:
  - includeDetails (boolean) para expandir objetos relacionados.
  - estado (string) para cambios de estado.

Integraciones (puntos críticos del flujo)
- userservice:
  - Verificación de existencia y obtención de rolId del usuario para autorizar acciones de negocio.
- propertyservice:
  - Verificación de existencia de propiedad.
  - “Disponibilidad” actualmente no está modelada; el cliente lo trata como “existe => disponible”.
- documentService:
  - Verificación de documentos aprobados como precondición de solicitud.
  - El cliente además contempla una operación de “conteo de aprobados” (dependencia contractual potencial).

Observabilidad y soporte
- Actuator presente en dependencias.
- Swagger/OpenAPI habilitado.
- Logs con bastante trazabilidad en los “clients” (útil para diagnosticar fallos de integración).

3.5 contactService (Mensajes de Contacto, Respuestas y Estadísticas)
--------------------------------------------------------------------
Responsabilidad principal
- Gestionar el “buzón” de contacto: recepción de mensajes, tracking por estado, respuesta administrativa y métricas.

Persistencia y modelo de datos
- Entidad principal: MensajeContacto.
- Campos típicos:
  - remitente (nombre, email, teléfono opcional)
  - asunto y mensaje
  - usuarioId opcional (si el remitente está autenticado)
  - estado (PENDIENTE / EN_PROCESO / RESUELTO)
  - respuesta, respondidoPor (adminId)
  - fechas de creación/actualización

Capacidades funcionales
- Crear mensaje:
  - Valida longitud del mensaje (mín 10, máx 5000).
  - Si hay usuarioId:
    - verifica que el usuario exista en userservice
    - limita mensajes pendientes por usuario: máximo 5
  - Si no hay usuarioId:
    - limita mensajes pendientes por email: máximo 5
  - Estado inicial: PENDIENTE y fechaCreacion asignada.
- Consultas:
  - Listado general (includeDetails opcional: adjunta UsuarioDTO consultado en userservice).
  - Consulta por ID.
  - Listado por email y por usuarioId.
  - Filtros por estado.
  - Listado de “sin responder” (mensajes pendientes sin respuesta).
  - Búsqueda por palabra clave (sobre asunto o mensaje).
- Gestión administrativa:
  - Cambiar estado del mensaje (valida enumeración de estados).
  - Responder mensaje:
    - valida que quien responde sea admin (consulta userservice y evalúa rol ADMIN)
    - guarda respuesta, respondidoPor y fechaActualizacion
    - permite setear nuevo estado o por defecto lo marca RESUELTO
  - Eliminar mensaje:
    - requiere adminId; valida que sea admin.
- Estadísticas:
  - Devuelve conteos por estado y total.

Métodos HTTP utilizados
- POST: crear mensaje; responder mensaje (con body JSON de respuesta).
- GET: listados, filtros, búsqueda y estadísticas.
- PATCH: cambio de estado.
- DELETE: eliminación (administrativa).

Formatos de entrada/salida
- JSON (application/json):
  - MensajeContactoDTO para creación/lectura.
  - RespuestaMensajeDTO para responder (incluye respuesta, respondidoPor y opcionalmente nuevoEstado).
- Respuestas:
  - DTOs JSON.
  - Estadísticas como objeto/mapa JSON clave->valor (conteos).

Integraciones
- userservice:
  - validar existencia de usuario cuando el mensaje declara usuarioId
  - validar rol admin para responder/eliminar
  - expandir datos de usuario (includeDetails)

Observabilidad y soporte
- Swagger/OpenAPI habilitado.
- Actuator health/info expuesto.

--------------------------------------------------------------------
3.6 reviewService (Reseñas, Tipos de Reseña y Promedios)
--------------------------------------------------------------------
Responsabilidad principal
- Permitir reseñas y valoraciones tanto a propiedades como a usuarios.
- Calcular promedios de puntaje por propiedad y por usuario reseñado.
- Moderación simple mediante estado de reseña (activa/baneada/oculta).

Persistencia y modelo de datos
- Entidades principales: Review, TipoResena.
- Review contiene:
  - usuarioId (autor)
  - propiedadId o usuarioResenadoId (uno de los dos)
  - puntaje (1..10)
  - comentario (10..500 si existe)
  - tipoResenaId (catálogo)
  - estado (ACTIVA/BANEADA/OCULTA), fechaResena, fechaBaneo

Capacidades funcionales
- Crear reseña:
  - Verifica que el usuario autor existe (userservice).
  - Valida que el rol del usuario puede crear reseñas (ARRIENDATARIO/PROPIETARIO/ADMIN).
  - Valida puntaje 1..10.
  - Valida tipoResenaId existente.
  - Valida “target”:
    - si es reseña a propiedad:
      - propiedad debe existir (propertyservice)
      - evita duplicados por (usuarioId, propiedadId)
      - impide que el propietario reseñe su propia propiedad (compara propietarioId de la propiedad)
    - si es reseña a usuario:
      - usuario reseñado debe existir (userservice)
      - evita duplicados por (usuarioId, usuarioResenadoId)
      - impide reseñarse a sí mismo
  - Estado inicial: ACTIVA; setea fechaResena.
- Consultas:
  - Listado general (includeDetails opcional).
  - Consulta por ID.
  - Listados por usuario autor, por propiedad, por usuario reseñado.
  - Cálculo de promedio por propiedad y por usuario reseñado (si no hay datos, retorna 0.0).
- Moderación:
  - Actualizar estado de reseña con validación (ACTIVA/BANEADA/OCULTA).
  - Si se banea y no existía fechaBaneo, la setea.
- Eliminación:
  - Borrado por ID (no se observa enforcement de rol en endpoint; la constante sugiere que solo ADMIN debería, pero esa validación no está aplicada en el service mostrado).

Métodos HTTP utilizados
- POST: creación de reseña.
- GET: listados/lecturas y promedios.
- PATCH: cambio de estado.
- DELETE: eliminación.

Formatos de entrada/salida
- JSON (application/json):
  - ReviewDTO: usuarioId, puntaje, tipoResenaId, comentario y uno de (propiedadId | usuarioResenadoId).
- Respuestas:
  - DTOs enriquecidos cuando includeDetails=true: adjunta UsuarioDTO (autor), PropiedadDTO (si aplica) y UsuarioDTO reseñado (si aplica), además de tipoResenaNombre.
  - Promedios como número (double).

Integraciones
- userservice: validación de existencia y obtención de datos del usuario (autor y reseñado).
- propertyservice: validación de propiedad y obtención de propietarioId.

Observabilidad y soporte
- Swagger/OpenAPI habilitado.
- Actuator health/info expuesto.

====================================================================
4) FORMATOS Y CONTRATOS DE DATOS (RESUMEN)
====================================================================
4.1 Formato predominante: JSON
- Todos los microservicios exponen/consumen JSON para entidades de dominio (usuarios, propiedades, documentos, solicitudes, reseñas, contacto).
- Validaciones declarativas:
  - @NotNull, @NotBlank, @Positive, @Min/@Max, @Size, @Email, @Pattern, etc.

4.2 Formato adicional: multipart/form-data
- Solo propertyservice, para upload de fotos:
  - entrada: archivo binario (MultipartFile) en multipart/form-data
  - validación: content-type permitido y tamaño máximo

4.3 Fechas
- userservice:
  - fechas como string “yyyy-MM-dd” en UsuarioDTO (entrada/salida).
- propertyservice:
  - LocalDate serializado como string “yyyy-MM-dd” para fecha de creación de propiedad.
- documentService/reviewService/applicationService:
  - uso de java.util.Date (timestamp lógico). DocumentService además configura formato/zone para JSON.

====================================================================
5) OBSERVACIONES TÉCNICAS (Riesgos y consideraciones operativas)
====================================================================
- Configuración local contiene credenciales en texto plano:
  - No se replican aquí por seguridad. Recomendación operativa: mover a secretos/variables de entorno en el despliegue.
- Seguridad HTTP:
  - userservice permite todas las requests en SecurityConfig; y los demás servicios no muestran enforcement de auth.
  - Hoy el “control” depende de reglas internas (rolId/rolNombre) y de que el cliente haga las validaciones.
- Contratos “implícitos” entre servicios:
  - Varios clientes construyen URLs y asumen estructuras de DTO externas (UsuarioDTO/PropiedadDTO).
  - Hay una operación en applicationService que presupone que documentService ofrece un conteo de documentos aprobados (capacidad contractual potencial).
- Storage de fotos:
  - propertyservice guarda la “url” como ruta local; en producción normalmente se requiere storage/URL pública (blob/CDN) y un endpoint de serving.
